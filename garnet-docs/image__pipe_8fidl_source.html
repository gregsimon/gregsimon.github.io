7 // ImagePipe is a mechanism for streaming shared images between a producer
8 // and a consumer which may be running in different processes.
9 //
10 // Conceptually, the image pipe maintains a table of image resources supplied
11 // by the producer into which graphical content may be stored as well as a
12 // presentation queue containing a sequence of images which the producer has
13 // asked the consumer to present.
14 //
15 // The presentation queue is initially empty.
16 //
17 // Each entry in the presentation queue consists of an image together with a
18 // pair of optional synchronization fences:
19 // - Acquire fence: signaled by the producer when the image is ready to be consumed
20 // - Release fence: signaled by the consumer when the image is free to be freed or
21 // modified by the producer
22 //
23 // The producer performs the following sequence of steps to present content:
24 // - Allocate and add some number of images (often 2 or 3) to the image pipe
25 // to establish a pool using |AddImage()|.
26 // - Obtain the next available image from the pool.
27 // - Ask the consumer to enqueue an image for presentation and provide fences
28 // using |PresentImage()|.
29 // - Start rendering the image.
30 // - Signal the image's acquire fence when rendering is complete.
31 // - Loop to present more image, listen for signals on release fences to recycle
32 // images back into the pool.
33 //
34 // The consumer performs the following sequence of steps for each image which
35 // is enqueued in the presentation queue:
36 // - Await signals on the image's acquire fence.
37 // - If the fence wait cannot be satisfied or if some other error is detected,
38 // close the image pipe.
39 // Otherwise, begin presenting the image's content.
40 // - Retire the previously presented image (if any) from the presentation queue
41 // and signal its release fence when no longer needed.
42 // - Continue presenting the same image until the next one is ready. Loop.
43 //
44 // If the producer wants to close the image pipe, it should:
45 // - Close its side of the connection.
46 // - Wait on all release fences for buffers that it has submitted with
47 // |PresentImage()|.
48 // - Proceed with resource cleanup.
49 //
50 // When the consumer detects the image pipe has closed, it should:
51 // - Stop using/presenting any images from the pipe.
52 // - Unmap all VMOs associated with the images in the pipe.
53 // - Close all handles to the VMOs.
54 // - Signal all release fences for presented and queued buffers.
55 // - Close all handles to fences.
56 // - Close its side of the connection.
57 //
58 // When either party detects that a fence has been abandoned (remotely closed
59 // without being signaled) it should assume that the associated image is in
60 // an indeterminate state. This will typically happen when the other party
61 // (or one of its delegates) has crashed. The safest course of action is to
62 // close the image pipe, release all resources which were shared with the
63 // other party, and re-establish the connection to recover.
64 interface ImagePipe {
65  // Adds an image resource to image pipe.
66  //
67  // The |memory| is the handle of a memory object which contains the image
68  // data. It is valid to create multiple images backed by the same memory object;
69  // they may even overlap. Consumers must detect this and handle it accordingly.
70  // The type of memory stored in the VMO is |memory_type| (e.g. GPU memory, host
71  // memory). The |memory_offset| indicates the offset within the memory object at
72  // which the image data begins.
73  //
74  // The following errors will cause the connection to be closed:
75  // - |image_id| is already registered
76  // - |image_info| represents a format not supported by the consumer
77  // - |memory| is not a handle for a readable VMO
78  // - the image data expected at |memory_offset| according to the |image_info|
79  // exceeds the memory object's bounds
80  AddImage(uint32 image_id, ImageInfo image_info,
81  handle<vmo> memory, MemoryType memory_type, uint64 memory_offset);
82 
83  // Removes an image resource from the pipe.
84  //
85  // The |image_id| is detached from the image resource and is free to be
86  // reused to add a new image resource.
87  //
88  // Removing an image from the image pipe does not affect the presentation
89  // queue or the currently presented image.
90  //
91  // The producer must wait for all release fences associated with the image to
92  // be signaled before freeing or modifying the underlying memory object since
93  // the image may still be in use in the presentation queue.
94  //
95  // The following errors will cause the connection to be closed:
96  // - |image_id| does not reference a currently registered image resource
97  RemoveImage(uint32 image_id);
98 
99  // Enqueues the specified image for presentation.
100  //
101  // The |acquire_fence|, if not null, specifies a fence which the consumer must
102  // await before presenting the image.
103  // The |release_fence|, if not null, specifies a fence which the consumer must
104  // signal when it is safe for the producer to free or modify the image.
105  // |presentation_time| specifies the time on or after which the
106  // client would like the enqueued operations should take visible effect
107  // (light up pixels on the screen), expressed in nanoseconds in the
108  // |CLOCK_MONOTONIC| timebase. Desired presentation times must be
109  // monotonically non-decreasing.
110  //
111  // |presentation_info| returns timing information about the submitted frame
112  // and future frames (see presentation_info.fidl).
113  //
114  // The following errors will cause the connection to be closed:
115  // - |image_id| does not reference a currently registered image resource
116  PresentImage(uint32 image_id, uint64 presentation_time,
117  handle<event>? acquire_fence, handle<event>? release_fence) =>
118  (PresentationInfo presentation_info);
119 };
