12  // Enqueue a list of operations, which will not take effect until Present() is
13  // called. It is allowable to call Enqueue() several times before calling
14  // Present(); the effect is the same as if all operations had been
15  // concatenated into a single Enqueue() call.
16  Enqueue(array<Op> op);
17 
18  // Present all previously enqueued operations. In order to pipeline the
19  // preparation of the resources required to render the scene, two lists of
20  // fences (implemented as events) are passed.
21  //
22  // SCHEDULING PRESENTATION
23  //
24  // |presentation_time| specifies the time on or after which the
25  // client would like the enqueued operations should take visible effect
26  // (light up pixels on the screen), expressed in nanoseconds in the
27  // |CLOCK_MONOTONIC| timebase. Desired presentation times must be
28  // monotonically non-decreasing.
29  //
30  // Using a desired presentation time in the present or past (such as 0)
31  // schedules enqueued operations to take visible effect as soon as possible
32  // (during the next frame to be prepared).
33  //
34  // Using a desired presentation time in the future schedules the enqueued
35  // operations to take visible effect as closely as possible to or after
36  // the stated time (but no earlier).
37  //
38  // Each rendered frame has a target presentation time. Before rendering
39  // a frame, the scene manager applies all enqueued operations associated
40  // with all prior calls to |Present()| whose desired presentation time
41  // is on or before the frame's target presentation time.
42  //
43  // The |Present()| method does not return until the scene manager begins
44  // preparing the first frame which includes its presented content.
45  // Upon return, the |PresentationInfo| provides timing information for the
46  // frame which includes the presented content.
47  //
48  // To present new content on each successive frame, wait for |Present()|
49  // to return before calling |Present()| again with content for the next
50  // frame.
51  //
52  // It is also possible to enqueue and present successive frames of content
53  // all at once with increasing desired presentation times, incrementing by
54  // |PresentationInfo.presentation_interval| for each one.
55  //
56  // Animation updates are also coordinated in terms of presentation time.
57  //
58  // TODO(jeffbrown): Defining presentation time in terms of |CLOCK_MONOTONIC|
59  // simplifies synchronization across subsystems but it might be too simple.
60  // We should consider using a synthetic timebase and describing its relation
61  // to other clocks separately. That would make it possible to present
62  // content (animations, media, and UI) in "slow mode" simply by varying the
63  // timing relation, assuming clients play along.
64  //
65  // SYNCHRONIZATION
66  //
67  // |acquire_fences| are used by Mozart to wait until all of the session's
68  // resources are ready to render (or to allow downstream components, such as
69  // the Vulkan driver, to wait for these resources).
70  //
71  // For example, Fuchsia's Vulkan driver allows an zx::event to be obtained
72  // from a VkSemaphore. This allows a Mozart client to submit a Vulkan command
73  // buffer to generate images/meshes/etc., and instructing Vulkan to signal a
74  // VkSemaphore when it is done. By inserting the zx::event corresponding to
75  // this semaphore into |acquire_fences|, the client allows Mozart to submit work
76  // to the Vulkan driver without waiting on the CPU for the event to be
77  // signalled.
78  //
79  // |release_fences| is a list of events that will be signalled by Mozart when
80  // the updated session state has been fully committed: future frames will be
81  // rendered using this state, and all frames generated using previous session
82  // states have been fully-rendered and presented to the display.
83  //
84  // Together, |acquire_fences| and |release_fences| are intended to allow clients
85  // to implement strategies such as double-buffering. For example, a client
86  // might do the following:
87  // 1) create two Image with resource IDs #1 and #2.
88  // 2) create two Materials with resource IDs #3 and #4, which respectively
89  // use Images #1 and #2 as their texture.
90  // 3) create a tree of Nodes and attach them to the scene.
91  // 4) set one of the nodes above, say #5, to use Material #3.
92  // 5) submit a Vulkan command-buffer which renders into Image #1, and
93  // will signal a VkSemaphore.
94  // 6) call Present() with one acquire-fence (obtained from the VkSemaphore
95  // above) and one newly-created release-fence.
96  //
97  // After the steps above, Mozart will use the committed session state to render
98  // frames whenever necessary. When the client wants to display something
99  // different than Image #1, it would do something similar to steps 4) to 6):
100  // 7) set Node #5 to use Material #4.
101  // 8) submit a Vulkan command-buffer which renders into Image #1, and
102  // will signal a VkSemaphore.
103  // 9) call Present() with one acquire-fence (obtained from the VkSemaphore
104  // above) and one newly-created release-fence.
105  //
106  // Finally, to continually draw new content, the client could repeat steps
107  // 4) to 9), with one important difference: step 5) must wait on the event
108  // signalled by step 9). Otherwise, it might render into Image #1 while that
109  // image is still being used by Mozart to render a frame. Similarly, step 8)
110  // must wait on the event signalled by step 6).
111  //
112  // The scenario described above uses one acquire-fence and one release-fence,
113  // but it is easy to imagine cases that require more. For example, in addition
114  // to using Vulkan to render into Images #1 and #2, the client might also
115  // upload other resources to Vulkan on a different VkQueue, which would
116  // would signal a separate semaphore, and therefore require an additional
117  // acquire-fence.
118  //
119  // Note: |acquire_fences| and |release_fences| are only necessary to synchronize
120  // access to memory (and other external resources). Any modification to
121  // resources made via the Session API are automatically synchronized.
122  Present(uint64 presentation_time,
123  array<handle<event>> acquire_fences, array<handle<event>> release_fences) =>
124  (PresentationInfo presentation_info);

126  // Performs a hit test along the specified ray.
127  //
128  // Returns a list of intersections of tagged nodes between the ray
129  // and the geometry of the subtree of nodes at |node_id|. Results are
130  // sorted by increasing distance from the ray's origin. Nodes which are
131  // hit at the same distance are sorted by the sequence they would appear
132  // in a post-order tree traversal of the scene. Equivalently, the
133  // results are ordered by decreasing hit specificity.
134  //
135  // The session must associate a tag with any nodes it would like to
136  // observe in the result set. Untagged nodes are still considered in
137  // hit tests and may cause hits to be reported for containing nodes
138  // which are tagged.
139  //
140  // The hit test only considers the currently presented content at
141  // the time of the request. This means that content pending presentation
142  // may not be considered for hit testing until the scene manager begins
143  // preparing a frame which includes it.
144  //
145  // |ray_origin| is a point in the node's local coordinate system
146  // from which the hit test ray originates.
147  //
148  // |ray_direction| is a vector in the node's local coordinate system
149  // which expresses how the ray propagates from the origin. It must
150  // be non-zero (otherwise nothing will be hit) but it does not need
151  // to be normalized.
152  //
153  // Returns a null hit array if |node_id| is unknown or not a node.
154  HitTest(uint32 node_id, vec3 ray_origin, vec3 ray_direction) =>
155  (array<Hit>? hits);
156 };
157 
158 // Describes where a hit occurred within the content of a node tagged
159 // by this session.
160 //
161 // To compute the point of intersection within the node's local coordinate
162 // system, perform the following calculation using the ray which was
163 // originally passed to |Session.HitTest()|.
164 //
165 // hit_point = ray.origin + (hit.distance * ray.direction)
166 // local_point = hit.inverse_transform * hit_point
167 struct Hit {
168  // The node's tag value.
169  uint32 tag_value;
170 
171  // The inverse transformation matrix which maps the coordinate system of
172  // the node at which the hit test was initiated into the local coordinate
173  // system of the node which was hit.
174  mat4 inverse_transform;
175 
176  // The distance from the ray's origin to the closest point of intersection
177  // in multiples of the ray's direction vector.
178  float distance;
179 };
